<objective>
## Goal
Eliminate the recurring Inngest summary-step 524 timeout by identifying its root cause with evidence, then applying a single targeted fix — not another reactive patch.

## Purpose

STATE.md explicitly flags this as the active concern. Until the timeout is root-caused, every paid analysis is a coin flip and the executive_summary / action_plan that users actually read will keep silently degrading to the fallback string.

- A reanalyzed real panel that completes end-to-end with a real Claude-generated action_plan (not the fallback).

<task type="checkpoint:decision" gate="blocking">
  <decision>Which fix strategy do we apply, given the diagnosis?</decision>
  <context>
    The diagnosis from Task 1 should point to one of three structural causes. Pick deliberately — this is the gate against another reactive patch.
  </context>
  <options>
    <option id="stream-sonnet">
      <name>Switch the summary Anthropic call to streaming (messages.stream) and aggregate the stream</name>
      <pros>
        - Keeps the connection emitting bytes continuously, defeating Cloudflare's 100s idle-timeout.
        - No model downgrade; output quality preserved.
        - Smallest diff; isolated to ~30 lines around the anthropic call.
        - Matches Anthropic's documented pattern for long-running generations.
      </pros>
      <cons>
        - Only addresses the gateway-idle case. If Anthropic itself is taking >300s, this won't save it.
        - Slightly more code to handle stream events vs single response.
      </cons>
    </option>
    <option id="split-step">
      <name>Split generate-summary into multiple step.run calls (fetch-context, claude-summary, store-plan, risk-calcs)</name>
      <pros>
        - Each step becomes small and individually retryable. Inngest will retry only the failed substep.
        - Easier observability per step.
        - Aligns with Inngest's intended usage pattern.
      </pros>
      <cons>
        - Doesn't actually shorten the Claude call itself — if that's the cause of 524, this only redistributes the failure.
        - Larger diff, more places for refactor mistakes.
      </cons>
    </option>
    <option id="bypass-cloudflare">
      <name>Point Inngest at the *.vercel.app domain (or upgrade Cloudflare timeout) instead of the proxied production domain</name>
      <pros>
        - If Cloudflare 100s is the confirmed cause, this removes it directly with zero code change.
        - Keeps the user-facing domain on Cloudflare for caching/analytics.
      </pros>
      <cons>
        - Infrastructure change; depends on how Inngest's serve URL is currently configured.
        - Doesn't help if Vercel's own gateway or Anthropic is the bottleneck.
        - Loses any Cloudflare protection on the Inngest endpoint specifically.
      </cons>
    </option>
  </options>
  <resume-signal>Select: stream-sonnet, split-step, bypass-cloudflare, or describe a combined approach</resume-signal>
</task>
